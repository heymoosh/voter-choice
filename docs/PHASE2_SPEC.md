# PHASE2_SPEC.md — Multilingual Extension (Spanish Language Support)

**Version:** 1.0
**Status:** Draft (Phase 0.2)
**Last updated:** March 13, 2026

This document describes the desired behavior and outcomes for adding Spanish language support to the ballot research tool. It is the shared modification request that all five workflow runs receive in Phase 2. It describes **what** should change, not **how** to implement it.

---

## Overview

Extend the existing ballot research tool to support both English and Spanish. A user can toggle between languages at any time. All user-facing text — UI labels, instructions, error messages, static content, and the customized AI prompt output — must be available in both languages.

The architecture should make adding a third language (e.g., Simplified Chinese, Vietnamese, Tagalog) straightforward — not requiring duplication of components, pages, or logic.

---

## Language Toggle

### Behavior

- A language toggle is visible on the page at all times, regardless of scroll position
- The toggle switches between **English** and **Español**
- Default language is English
- Switching languages updates ALL user-facing text on the page immediately, without a page reload and without losing application state (e.g., if a zip code has been submitted and results are displayed, they remain displayed in the new language)
- The toggle is accessible via keyboard and announced to screen readers
- The selected language persists if the user refreshes the page (use browser storage, not a URL parameter or server state)

### Placement

- The toggle should be located in a conventional, easily discoverable position (e.g., top-right of the page or within a header/nav area)
- It must not interfere with or obscure existing UI elements
- On mobile, it must remain accessible without requiring a menu to open

### Required `data-testid`

| `data-testid` | Element | Purpose |
|----------------|---------|---------|
| `language-toggle` | The language switch control | E2e tests click to change language |

## What Gets Translated

### All UI Text

Every piece of user-facing text on the page must be available in both languages. This includes:

- Hero section headline and subtitle
- Chatbot names and links labels (the chatbot names themselves — Claude, ChatGPT, etc. — stay in English as they are proper nouns)
- Zip code input label and placeholder text
- Submit button text
- All error messages (validation errors, zip not found, no upcoming election, etc.)
- State info card labels ("Election," "Registration deadlines," "Early voting," etc.)
- Deadline status indicators ("12 days left," "Passed," etc.)
- State selector prompt ("This zip code spans multiple states. Which state are you voting in?")
- Copy button text ("Copy to Clipboard" / "Copied!")
- Instructions above the prompt output
- Tips section content
- Footer text
- Accessibility text (skip-to-content link, ARIA labels, screen reader announcements)

### State Election Data Display

- **Labels** for data fields (e.g., "Election date," "Voter ID required") are translated
- **Data values** from the JSON (e.g., state names, election names, accepted ID types, voting rule details) remain in English for Phase 2. Translating 50 states of election-specific data is a data quality issue, not a UI issue — it can be done later without changing the architecture
- **Exception:** The `earlyVoting.notes` and `votingRules.phonesAtPollsDetail` fields contain human-readable sentences. These remain in English for Phase 2 but should be structured so they *could* be translated later (i.e., the display layer should not assume these fields are always in the UI language)

### The Customized AI Prompt Output

This is the core product — the prompt the user copies and pastes into a chatbot. It has two parts:

#### Part 1: The Main Prompt (from BALLOT_PROMPT.md)

- In **English mode:** The prompt is served in English (the original `BALLOT_PROMPT.md` text)
- In **Spanish mode:** The prompt is served in Spanish — a complete, natural Spanish translation of the full prompt
- The Spanish translation must read as fluent, natural Spanish — not machine-translation artifacts. It should use "tú" (informal) voice to match the conversational, approachable tone of the English original
- The translation includes all formatting, structure, and instructions from the original prompt
- The Spanish prompt must be stored as a complete translation, not generated by string interpolation on fragments. Prompts are long-form prose with voice and tone — piecemeal assembly produces unnatural text

#### Part 2: The Pre-filled Context Block

- In **English mode:** The context block is in English (as currently specified in PROJECT_SPEC.md)
- In **Spanish mode:** The context block structure and labels are in Spanish, but injected data values (state name, election name, URLs, dates) remain in their original form from the JSON
- Example Spanish context block:

```
¡Hola! Voy a votar en **Texas**. Mi código postal es **73301**.

Esto es lo que sé sobre mi próxima elección:
- **Elección:** 2026 Texas Primary Election el 3 de marzo de 2026
- **Tipo de elección:** Primaria (abierta)
- **Fechas límite de registro:** En línea antes del 2 de febrero de 2026, por correo antes del 2 de febrero de 2026 (fecha de matasellos), en persona antes del 2 de febrero de 2026
- **Votación anticipada:** Del 17 de febrero al 28 de febrero de 2026
- **Identificación para votar:** Requerida. [accepted IDs in English]
- **Teléfonos en las casillas:** [policy detail in English]
- **Mi boleta de muestra:** [sampleBallotLookup URL]
- **Mi oficina electoral del condado:** [countyElectionLookup URL]

Ayúdame con mi boleta.
```

### Tips Section

- The tips section content is translated to Spanish
- Tips that reference the AI chatbot interaction (e.g., "You can say 'I don't know'") should be translated naturally, noting that the user may be interacting with the chatbot in either language

### Footer

- All footer text is translated
- The attribution line translates to: "Creado por una persona usando herramientas de IA"

## What Does NOT Change

- Application logic (zip code lookup, deadline calculation, state data resolution)
- Data model or JSON schema — no new fields are added to the election data
- Page layout or section order
- Existing `data-testid` attributes (all Phase 1 test IDs remain unchanged and functional)
- The set of supported states in stub data
- Any Phase 1 acceptance criteria — everything that worked before must still work

---

## Date Formatting

- In English mode: dates display in U.S. format (e.g., "March 3, 2026")
- In Spanish mode: dates display in Spanish format (e.g., "3 de marzo de 2026")
- Date calculations (days remaining, deadline status) are language-independent — only the display label changes ("12 days left" → "Quedan 12 días")

## Error Messages in Spanish

All error messages have Spanish equivalents:

| English | Spanish |
|---------|---------|
| "Please enter a zip code" | "Por favor ingresa un código postal" |
| "Please enter a valid 5-digit zip code" | "Por favor ingresa un código postal válido de 5 dígitos" |
| "We don't have data for this zip code yet..." | "Aún no tenemos datos para este código postal..." |
| "This zip code spans multiple states. Which state are you voting in?" | "Este código postal abarca varios estados. ¿En qué estado vas a votar?" |
| "Registration deadlines for this election have passed..." | "Las fechas límite de registro para esta elección ya pasaron..." |
| "No upcoming elections found for [State]..." | "No se encontraron elecciones próximas para [State]..." |

These are provided as reference translations — the workflow may organize and store them however it chooses.

## Accessibility Requirements (Additions to Phase 1)

- The language toggle is keyboard-navigable and operable via Enter/Space
- Switching language updates the page's `lang` attribute (e.g., `lang="en"` → `lang="es"`)
- Screen readers announce the language change
- All translated text maintains WCAG AA color contrast
- Translated text that is longer than the English equivalent (common in Spanish) must not break layouts — the UI must accommodate text expansion gracefully

## Acceptance Criteria

A Phase 2 run is "done" when ALL Phase 1 acceptance criteria still pass AND the following are true:

### Language Toggle

- [ ] Language toggle is visible and accessible at all times
- [ ] Clicking/activating the toggle switches all UI text between English and Spanish
- [ ] Language preference persists across page refreshes
- [ ] Switching language does not reset application state (zip code results remain visible)
- [ ] The `language-toggle` `data-testid` is present on the toggle element

### Translated Content

- [ ] All UI labels, instructions, error messages, and static content are available in both languages
- [ ] The full AI prompt (Part 1 — main prompt from BALLOT_PROMPT.md) is available in fluent Spanish
- [ ] The pre-filled context block (Part 2) generates correctly in both languages
- [ ] Date formatting follows language conventions (English vs. Spanish format)
- [ ] Deadline status indicators display in the selected language
- [ ] Tips section content is translated
- [ ] Footer text is translated

### Data Handling

- [ ] State election data values (from JSON) display correctly regardless of language
- [ ] Injected data in the prompt context block works correctly in both languages
- [ ] No data or logic regressions from Phase 1

### Accessibility

- [ ] The page `lang` attribute updates when language changes
- [ ] Language toggle is keyboard-accessible
- [ ] Longer Spanish text does not break layouts at any breakpoint (mobile, tablet, desktop)
- [ ] All Phase 1 accessibility requirements still pass

### Architecture

- [ ] Adding a third language would require only new translation content, not structural changes to components or logic
- [ ] Translation content is separated from component code (not hardcoded strings in JSX/TSX)

## Out of Scope

These are explicitly NOT part of Phase 2:

- Translating election data values (state names, election names, ID types, voting rule details)
- Auto-detecting browser language
- URL-based language routing (e.g., `/es/`)
- Right-to-left (RTL) language support
- More than two languages (the architecture should support it; the implementation does not need to)
- Translating the chatbot names (Claude, ChatGPT, etc. — proper nouns)
- Any changes to the data model or JSON schema

## Acceptance Criteria

Each criterion is a single testable assertion. Every e2e and integration test should reference the `AC-2.x` ID it covers in its `describe()` or `test()` name.

- **AC-2.1** — A visible keyboard-accessible language toggle switches the page between English and Spanish.
- **AC-2.2** — Switching language updates all user-facing labels, instructions, static content, and error text without resetting the current zip-code results.
- **AC-2.3** — The selected language persists across refreshes.
- **AC-2.4** — The full ballot-research prompt is available in fluent Spanish, and the pre-filled context block renders correctly in the selected language.
- **AC-2.5** — Date formatting and deadline-status messaging follow the selected language conventions.
- **AC-2.6** — The Phase 2 `language-toggle` test id is present and all Phase 1 test ids remain intact.
- **AC-2.7** — The translation architecture isolates content from component logic so a third language can be added without duplicating the UI structure.

## Required UX Flows

- **UX-2.1** — A voter enters a zip code in English, switches to Spanish after results load, and sees the same election data and prompt content preserved in Spanish.
- **UX-2.2** — A voter chooses Spanish, refreshes the page, and returns to the same language preference.
- **UX-2.3** — A keyboard-only user toggles languages and receives the updated `lang`/screen-reader context without losing focus control.

## Non-Functional Requirements

- **NFR-2.1** *(Performance)* — First Load JS must remain at or below 130 kB after adding Spanish translations.
- **NFR-2.2** *(Accessibility)* — Lighthouse accessibility score must be at least 90 with translated content enabled.
- **NFR-2.3** *(Performance)* — Lighthouse performance score must remain at or above 90.
- **NFR-2.4** *(Security)* — `npm audit --json` must report 0 HIGH or CRITICAL vulnerabilities.
- **NFR-2.5** *(UX)* — Longer Spanish strings must not cause layout breakage at mobile, tablet, or desktop breakpoints.
