# Research: Spanish Language Support

**Branch**: `003-spanish-language-support` | **Date**: 2026-04-03

## Phase 0 Research Findings

No NEEDS CLARIFICATION items from Technical Context — all technology choices are
determined by the existing scaffold. Research focuses on i18n patterns for Next.js 15
App Router with React 19 and the specific requirements from PHASE2_SPEC.md.

---

### Decision 1: i18n Architecture — React Context vs. URL-based routing

**Decision**: React Context with `localStorage` persistence (client-side, no URL routing)

**Rationale**: PHASE2_SPEC.md explicitly rejects URL-based routing (`/es/`). The tool
is a static single-page app. React Context is the correct pattern for client-side
language state that needs to be shared across all components without URL changes.
Next.js 15 App Router supports this via a `'use client'` context provider wrapping
the page content (server component is a thin shell).

**Alternatives considered**:
- `next-intl` / `react-i18next` libraries: Overkill for 2 languages; adds bundle size
  and configuration complexity. Custom typed translation interface is leaner.
- URL routing (`/es/`): Explicitly out of scope per PHASE2_SPEC.md.
- `i18n` routing via `next.config.ts`: Same problem as URL routing.

**Pattern**: `LanguageProvider` wraps page content. `useLanguage()` hook returns
`{ lang, setLang }`. Components consume translations via `const t = useTranslations()`.

---

### Decision 2: Translation Storage — Typed Interface vs. JSON Files

**Decision**: Typed TypeScript module (`src/lib/translations.ts`) with inline objects

**Rationale**: With exactly 2 languages and ~50 string keys, a TypeScript file with
typed interfaces provides compile-time safety (missing translation key = type error)
at negligible cost. JSON files would require an import step and lose type inference.
For interpolated strings (e.g., "No upcoming elections found for {state}"), TypeScript
functions are cleaner than placeholder token patterns.

**Alternatives considered**:
- JSON files per language: Loses type safety, requires runtime validation.
- Single JSON with nested language keys: Same type-safety issue.
- i18next JSON format: Unnecessary abstraction for 2 languages.

**Pattern**:
```typescript
type Language = "en" | "es";

interface Translations {
  hero: { title: string; subtitle: string; };
  zipForm: { label: string; placeholder: string; submit: string; };
  errors: {
    empty: string;
    invalid: string;
    notFound: string;
    noElection: (stateName: string) => string;
    multiState: string;
  };
  // ... all UI sections
}

const en: Translations = { ... };
const es: Translations = { ... };
const translations: Record<Language, Translations> = { en, es };
```

### Decision 3: Language Persistence — localStorage vs. sessionStorage vs. Cookie

**Decision**: `localStorage` with graceful fallback (try/catch)

**Rationale**: `localStorage` persists across sessions (user closes/reopens browser).
`sessionStorage` only persists for the tab lifetime — not what PHASE2_SPEC.md requires.
Cookies add server-side complexity inappropriate for a static app. The PHASE2_SPEC.md
Assumptions section explicitly confirms localStorage. Wrap access in try/catch for
private-mode compatibility.

**Key**: `localStorage.setItem('ballot-tool-lang', 'es')` / `getItem('ballot-tool-lang')`

---

### Decision 4: SSR Hydration Guard

**Decision**: Initialize language state to `'en'` on server; read localStorage in
`useEffect` to avoid hydration mismatch.

**Rationale**: Next.js App Router renders server-side HTML. If the server renders "en"
and the client has "es" stored, React hydration will mismatch. The standard pattern:
start with `'en'` (matches server output), then in `useEffect` read localStorage and
update to `'es'` if stored. This produces a brief flash to English before switching —
acceptable for a language toggle (same pattern used in Superpowers Phase 2 run).

---

### Decision 5: LanguageToggle Placement

**Decision**: Fixed top-right position (`fixed top-4 right-4 z-50`)

**Rationale**: PHASE2_SPEC.md requires the toggle to be "visible at all times,
regardless of scroll position." Fixed positioning is the only reliable cross-device
way to achieve this. Top-right is the conventional placement for language toggles
(matches PHASE2_SPEC.md suggested placement). `z-50` ensures it doesn't get covered
by modals.

---

### Decision 6: Spanish Prompt Storage

**Decision**: Complete `BALLOT_PROMPT_ES` constant in `generatePrompt.ts` (not fragments)

**Rationale**: PHASE2_SPEC.md explicitly requires: "The Spanish translation must be
stored as a complete translation, not generated by string interpolation on fragments.
Prompts are long-form prose with voice and tone — piecemeal assembly produces unnatural
text." The prompt is ~200 lines; storing it as a complete constant matches the existing
`BASE_PROMPT` pattern.

---

### Decision 7: generatePrompt Language Parameter

**Decision**: Add optional `lang: Language = 'en'` parameter to `generatePrompt()`

**Rationale**: The function already accepts `todayISO` as an optional injection
parameter for testability. Adding `lang` follows the same pattern. The function
selects `BALLOT_PROMPT_ES` vs `BASE_PROMPT` and builds the Spanish vs English
context block based on `lang`. No structural change to `CustomizedPrompt` type needed.

---

### Decision 8: Date Formatting for Spanish

**Decision**: Use `Intl.DateTimeFormat` with `locale: 'es-US'` and `dateStyle: 'long'`

**Rationale**: `Intl.DateTimeFormat('es-US', { dateStyle: 'long' })` produces
"3 de marzo de 2026" natively in all modern browsers. The existing `getDeadlineStatus`
function generates the `label` string. Add an optional `locale` parameter, defaulting
to `'en-US'`. Pure function, easy to test with injected values.

---

### Decision 9: page.tsx Refactor for Server/Client Split

**Decision**: Refactor `page.tsx` as thin server component wrapping `LanguageProvider`
and a new `PageContent` client component.

**Rationale**: The existing `page.tsx` has all content hardcoded as a server component.
For language switching to work, the hero section, tips, and footer must be client
components that consume language context. Pattern: `page.tsx` (server) renders
`<LanguageProvider><LanguageToggle /><PageContent /></LanguageProvider>`. This is
consistent with Next.js App Router best practices and matches the approach used in
other Phase 2 runs.
