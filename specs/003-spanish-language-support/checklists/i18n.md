# i18n Requirements Quality Checklist: Spanish Language Support

**Purpose**: Validate completeness, clarity, consistency, and measurability of
internationalization requirements for the Spanish language support feature.
Tests the requirements themselves — not the implementation.

**Created**: 2026-04-03
**Feature**: [spec.md](../spec.md)
**Focus**: Accessibility requirements, error state completeness, data model coverage

---

## Requirement Completeness

- [ ] CHK001 Are translation requirements specified for ALL user-facing strings, including
  dynamic/interpolated strings (e.g., "No upcoming elections found for {state}")? [Completeness, Spec §FR-006]
- [ ] CHK002 Are requirements defined for the behavior when a translation key is missing at runtime
  (fallback strategy documented)? [Completeness, Spec §Edge Cases]
- [ ] CHK003 Are requirements specified for which data values remain untranslated in Spanish mode
  (state names, election names, accepted IDs, URLs, notes fields)? [Completeness, Spec §FR-017]
- [ ] CHK004 Are requirements defined for the language toggle's visual state (what it displays in
  each language mode — "Español" vs. "English")? [Completeness, Spec §FR-001]
- [ ] CHK005 Are requirements defined for language behavior in the multi-state selector flow
  (Spanish prompt text when zip spans multiple states)? [Completeness, Spec §FR-006]
- [ ] CHK006 Are accessibility text requirements specified for the language toggle
  (aria-label, screen reader announcement on activation)? [Completeness, Spec §FR-001]

---

## Requirement Clarity

- [ ] CHK007 Is "fluent Spanish translation" defined with measurable or verifiable criteria
  beyond a subjective quality judgment? [Clarity, Spec §FR-007]
- [ ] CHK008 Is "immediately" in "switches all UI text immediately" (language switch) quantified
  with a specific timing threshold? [Clarity, Spec §SC-002]
- [ ] CHK009 Is "text expansion" tolerance specified — are there concrete max-width or overflow
  requirements for translated strings that are longer than English? [Clarity, Spec §FR-013]
- [ ] CHK010 Is "fixed position" for the language toggle specified precisely enough to differentiate
  from `position: sticky` or `position: absolute`? [Clarity, Spec §FR-001]
- [ ] CHK011 Is the "tú" (informal) voice requirement for the Spanish prompt described in a way
  that is verifiable without Spanish language expertise? [Clarity, Spec §FR-007]
- [ ] CHK012 Are "Spanish labels" for the context block specified with example translations,
  or only described abstractly? [Clarity, Spec §FR-008]

---

## Requirement Consistency

- [ ] CHK013 Is the language toggle placement requirement in the spec (§FR-001) consistent with
  the UI contract (fixed top-right) and the constitution (Principle VI)? [Consistency]
- [ ] CHK014 Are error message Spanish translations in the spec consistent with those shown
  in the PHASE2_SPEC.md reference translation table? [Consistency, Spec §US3]
- [ ] CHK015 Does the spec's treatment of data values (remain English) align consistently across
  all components: state info card, context block, and error messages? [Consistency, Spec §FR-008, FR-006]
- [ ] CHK016 Are date formatting requirements consistent between the spec (§FR-009), the data-model
  (getDeadlineStatus locale param), and the context block example in PHASE2_SPEC.md? [Consistency]
- [ ] CHK017 Does the Phase 2 acceptance criteria (§SC-001 — 42 e2e tests pass) align with the
  Phase 1 requirement that 6 pre-existing failures on the vanilla branch were known issues?
  (This branch should have 42/42, not 36/42.) [Consistency, Assumption]

## Acceptance Criteria Quality

- [ ] CHK018 Is SC-003 ("100% of user-facing string keys have Spanish translations") verifiable
  without manual inspection of all strings? Is a programmatic check defined? [Measurability, Spec §SC-003]
- [ ] CHK019 Is SC-004 ("Spanish prompt covers 100% of English prompt content") verifiable?
  Is "covers" defined as character-level, structural, or semantic equivalence? [Measurability, Spec §SC-004]
- [ ] CHK020 Is SC-007 ("≤ 30 kB bundle increase") tied to a specific measurement tool and
  baseline value? [Measurability, Spec §SC-007]
- [ ] CHK021 Is SC-008 ("adding third language requires only content changes") verifiable at spec
  level, or does it require code review to confirm? [Measurability, Spec §SC-008]
- [ ] CHK022 Is SC-002 ("all text updates visible within 200ms") achievable in worst-case mobile
  hardware scenarios? Is the measurement baseline defined? [Measurability, Spec §SC-002]

## Accessibility Requirements Coverage

- [ ] CHK023 Are requirements defined for how screen readers announce the language change when
  the toggle is activated (e.g., aria-live region, lang attribute, or separate announcement)? [Coverage, Spec §US5]
- [ ] CHK024 Are requirements specified for the skip-to-content link in Spanish mode — both the
  translation of the link text AND whether the link target changes? [Coverage, Spec §FR-006]
- [ ] CHK025 Are color contrast requirements explicitly maintained for translated Spanish text,
  given that Spanish text may render differently in some font metrics? [Coverage, Spec §FR-012]
- [ ] CHK026 Are requirements defined for WCAG AA compliance for the language toggle itself —
  contrast ratio, minimum touch target size, focus indicator? [Coverage, Spec §FR-001]
- [ ] CHK027 Are requirements defined for the `lang` attribute update timing — must it update
  synchronously with the text switch, or is a brief lag acceptable? [Coverage, Spec §FR-011]

## Error State Coverage

- [ ] CHK028 Are error message requirements defined for ALL 5 error states (empty, invalid, not-found,
  multi-state, no-election) in Spanish? [Completeness, Spec §US3, FR-006]
- [ ] CHK029 Are requirements defined for whether active error messages translate immediately
  when the language is switched while an error is displayed? [Coverage, Gap]
- [ ] CHK030 Is the exact Spanish translation for the "no-election" interpolated message specified
  ("No se encontraron elecciones próximas para [State]...")? [Clarity, Spec §US3]
- [ ] CHK031 Are requirements defined for loading state text translation? [Completeness, Spec §FR-006]

---

## Data Model Coverage

- [ ] CHK032 Is the `Language` type scope defined — are additional languages beyond "en" and "es"
  explicitly ruled out for Phase 2 without requiring a schema migration? [Coverage, Spec §FR-015]
- [ ] CHK033 Are requirements defined for the `Translations` interface structure — specifically,
  are function-typed keys (for interpolated strings) specified vs. plain strings? [Clarity, Spec §data-model]
- [ ] CHK034 Are requirements defined for how `CustomizedPrompt` type handles language — does the
  type need a `lang` field, or is it language-opaque? [Coverage, Gap]
- [ ] CHK035 Are requirements defined for the behavior when localStorage is unavailable
  (private browsing, storage disabled) — is English fallback specified and its UX described? [Coverage, Spec §Edge Cases]

---

## Non-Functional Requirements

- [ ] CHK036 Are performance requirements defined for the Spanish prompt generation specifically,
  given `BALLOT_PROMPT_ES` (~200 lines) is concatenated client-side? [Coverage, Gap]
- [ ] CHK037 Is bundle size impact explicitly bounded for the Spanish prompt constant (~200 lines
  of prose) — is this included in the ≤ 30 kB SC-007 target? [Completeness, Spec §SC-007]
- [ ] CHK038 Are requirements defined for server-side rendering behavior — specifically, is the
  brief flash to English (SSR hydration) before localStorage is read considered acceptable UX? [Clarity, Spec §Assumptions]

## Dependencies & Assumptions

- [ ] CHK039 Is the assumption that `Intl.DateTimeFormat` produces correct output for all
  targeted browsers documented and validated? [Assumption, Spec §research.md Decision 8]
- [ ] CHK040 Is the assumption that `localStorage` is the correct persistence mechanism validated
